## 2. LLM Inference — vLLM 0.18+ / Внешние API

| | |
|---|---|
| **Что** | Production-grade сервер для запуска LLM моделей (v0.18.0, март 2026) |
| **Зачем** | Генерация текста (ответы на вопросы с контекстом из RAG) |
| **Новое в 0.18** | gRPC serving, FlashAttention 4, PyTorch 2.10, elastic expert parallelism, KV cache offloading |
| **Почему не Ollama** | vLLM в **2–24 раза быстрее** (PagedAttention, continuous batching), multi-GPU, квантизация (GPTQ, AWQ, FP8, INT8) |

### Поддержка провайдеров

Поскольку vLLM использует **OpenAI-совместимый API**, наш LLM-клиент работает с **любым** провайдером через один и тот же протокол. Меняется только `base_url` и `api_key`:

| Провайдер | `base_url` | Когда использовать |
|---|---|---|
| **vLLM (локально)** | `http://localhost:8001/v1` | Production-сервер с GPU |
| **OpenAI** | `https://api.openai.com/v1` | Максимальное качество (GPT-4o) |
| **Anthropic** | Через адаптер | Claude для длинных документов |
| **Groq** | `https://api.groq.com/openai/v1` | Быстрая генерация |
| **Together AI** | `https://api.together.xyz/v1` | Доступ к open-source моделям |
| **Любой совместимый** | Пользовательский URL | Свои серверы, другие провайдеры |

### Ключевые преимущества vLLM для продакшена:

| Фича | Что даёт |
|---|---|
| **PagedAttention** | Эффективное управление GPU-памятью, больше concurrent запросов |
| **Continuous batching** | Обрабатывает новые запросы не дожидаясь окончания текущих |
| **FlashAttention 4** | Новое поколение attention — ещё быстрее |
| **gRPC serving** | Высокопроизводительный RPC помимо HTTP |
| **Tensor parallelism** | Раскидывает модель на несколько GPU |
| **Квантизация** | GPTQ, AWQ, FP8, INT8, AutoRound — большие модели на меньшем VRAM |

### Запуск LLM-модели:
```bash
vllm serve Qwen/Qwen2.5-7B-Instruct \
  --port 8001 \
  --dtype auto \
  --max-model-len 8192
```

### Запуск Embedding-модели:
```bash
vllm serve BAAI/bge-m3 \
  --port 8002 \
  --task embed \
  --dtype auto
```

> **Важно:** vLLM запускается как **отдельный процесс** (или два — для LLM и Embeddings). RAG-LLM общается с ними по HTTP через OpenAI-совместимый API.

## 3. Embedding-модель — BGE-M3 (через vLLM)

| | |
|---|---|
| **Что** | Модель, превращающая текст в числовой вектор |
| **Зачем** | Сравнение текстов по смыслу: «вопрос ↔ чанк документа» |
| **Какая модель** | `BAAI/bge-m3` — мультиязычная (100+ языков, включая русский), 1024-мерные вектора, до 8192 токенов на вход |
| **Почему bge-m3** | Лучшая мультиязычная embedding модель. Поддерживает dense + sparse + multi-vector retrieval. Отлично работает с русским текстом |
| **Альтернативы** | `nomic-embed-text` (слабее для русского), `e5-mistral-7b-instruct` (качественнее, но 7B параметров — тяжёлый) |

### Как работает:

```
Текст: "Retrieval-Augmented Generation позволяет..."
    → vLLM /v1/embeddings  (или OpenAI /v1/embeddings)
    → [0.023, -0.156, 0.891, ..., 0.042]  (1024 числа)
    → сохраняется в ChromaDB
```

## 4. Парсинг документов — Docling 2.80+ (IBM)

| | |
|---|---|
| **Что** | AI-библиотека для парсинга документов от IBM Research (v2.80.0) |
| **Зачем** | Вытащить текст из PDF + нарезать на чанки |
| **Новое** | **Heron** layout model (быстрее), VLM (Granite-Docling-258M), XBRL/WebVTT парсинг, извлечение изображений |
| **Почему не PyMuPDF** | Docling **на порядок умнее**: layout analysis, таблицы, OCR, формулы. PyMuPDF — просто текст |

### Что Docling умеет и зачем это нужно:

| Возможность | Что даёт |
|---|---|
| **Layout analysis** | Понимает колонки, заголовки, сноски — не путает порядок чтения |
| **Table recognition** | Таблицы парсит в структурированный формат (не кашу текста) |
| **OCR** | Работает со сканированными PDF (чего PyMuPDF не умеет) |
| **Formula extraction** | Математические формулы → LaTeX |
| **Multi-format** | PDF, DOCX, PPTX, XLSX, HTML, изображения |

```
pip install docling>=2.80.0
```

## 5. Chunking — Docling HybridChunker (встроенный)

| | |
|---|---|
| **Что** | Встроенный чанкер Docling |
| **Зачем** | Нарезка текста на оптимальные куски для RAG |
| **Почему встроенный** | HybridChunker **учитывает структуру документа**: заголовки, абзацы, таблицы. Не режет посередине предложения или таблицы |
| **Почему не LangChain** | LangChain RecursiveCharacterTextSplitter не знает структуру документа — режет вслепую по символам. Docling видит layout |

### Как работает HybridChunker:

```
PDF страница:
┌────────────────────────┐
│ Глава 3: Нейросети      │  ← заголовок
│                         │
│ Нейронные сети — это... │  ← абзац 1 (чанк 1)
│ ... вычислительные      │
│ модели, которые...      │
│                         │
│ 3.1 Архитектуры         │  ← подзаголовок
│                         │
│ Существует несколько... │  ← абзац 2 (чанк 2)
│ ┌──────┬──────┐         │
│ │ CNN  │ 1989 │         │  ← таблица (чанк 3, целиком)
│ │ RNN  │ 1997 │         │
│ └──────┴──────┘         │
└────────────────────────┘
```

Каждый чанк получает **метаданные**: заголовок раздела, номер страницы, тип элемента.

> **Плюс:** Отдельная библиотека для чанкинга **не нужна** — Docling решает и парсинг, и чанкинг из коробки. Парсер знает структуру → чанкер использует эту структуру.

## 6. Векторная база — ChromaDB 1.0+

|---|---|

## 7. Мониторинг ресурсов — psutil + pynvml

| Пакет | Что даёт |
|---|---|
| `psutil` | CPU % (по ядрам), RAM (использовано/всего), диск, сеть |
| `pynvml` | GPU: нагрузка %, температура, VRAM (использовано/всего), мощность |

Мониторинг привязан к **этапам пайплайна** — не просто "CPU 80%", а "CPU 80% **во время embedding генерации**".

| | |
|---|---|
| **Что** | Библиотека структурного логирования |
| **Зачем** | Каждый шаг пайплайна записывается: timestamp, длительность, что зашло, что вышло |

### Пример лога:

```

12:03:05.612 | INFO  | index     | ChromaDB: сохранено в коллекцию 'my_book'
```

## Чего НЕТ и почему

| Технология | Почему нет |
|---|---|
| **Ollama** | Слишком медленный для production, нет continuous batching |
| **LangChain** | Скрывает детали — противоречит цели прозрачности |
| **PyMuPDF** | Docling качественнее: layout, таблицы, OCR |
| **Отдельный chunker** | Docling HybridChunker уже встроен и знает структуру |
| **Docker** | Добавим позже. На первом этапе — напрямую |
| **PostgreSQL** | ChromaDB + файловая система достаточно |
