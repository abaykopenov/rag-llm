# 📋 RAG-LLM — Описание проекта

## Что это

> **Одна система — любой LLM:** работает с локальными моделями (vLLM) и облачными API (OpenAI, Anthropic, Groq, любой OpenAI-совместимый).

**RAG-LLM** — локальный сервис для работы с документами через LLM, в котором **видно, как именно система думает** на уровне пайплайна.

Это не «чат с PDF». Это **прозрачный RAG-движок**, который можно подключить к любому проекту через API и получать качественные ответы по загруженным документам.

---

## Проблема

При построении RAG-систем главная боль — **непрозрачность**:

- PDF распарсился криво → чанки получились мусорные → ответ слабый
- Embedding-модель подобрана неправильно → retrieval не находит нужное
- Prompt собран плохо → LLM галлюцинирует
- Всё это запущено **отдельно**: Ollama, ChromaDB, мониторинг, приложение — нигде нет единой картины

**Невозможно понять, где именно ломается**, не проверив каждый этап вручную.

---

## Решение

RAG-LLM объединяет весь пайплайн в **одном сервисе** и делает каждый этап **видимым**:

```
Загрузка PDF
     ↓
Парсинг (Docling) → видно извлечённый текст, таблицы, структуру
     ↓
Нарезка на чанки → видно каждый чанк, его длину, страницу, раздел
     ↓
Embeddings (vLLM / API) → вектора сохраняются в ChromaDB
     ↓
Вопрос пользователя
     ↓
Retrieval → видно какие чанки найдены, с какими scores
     ↓
Prompt → видно что именно ушло в модель
     ↓
LLM ответ (vLLM / OpenAI / Anthropic) → видно время генерации, токены
     ↓
Мониторинг → CPU, RAM, GPU на каждом этапе
```

## Как подключается к другим сервисам

```
Внешний сервис → POST /api/query
                 { "question": "...", "collection": "my_book" }

RAG-LLM       → JSON ответ:
                 {
                   "answer": "...",
                   "sources": [...],
                   "chunks_used": [...],
                   "scores": [...],
                   "timing": { "parse_ms": 45, "retrieval_ms": 23, "llm_ms": 2100 }
                 }
```

Потребитель берёт `answer` — готовый текст. Если нужна отладка — смотрит `chunks_used`, `scores`, `timing`.

---

## Ключевые принципы

1. **Прозрачность** — каждый этап виден и логируется
2. **API-first** — основной интерфейс REST API, UI опционален
3. **Production-ready** — vLLM, поддержка GPU, расчёт на серверы
4. **Multi-provider** — локальные LLM (vLLM) + облачные API (OpenAI, Anthropic, Groq, любой совместимый)
5. **Без магии** — никакого LangChain, все этапы контролируемы
6. **Встраиваемость** — подключается к любому проекту одним HTTP-запросом
